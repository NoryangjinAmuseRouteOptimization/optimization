# 솔버 베이스라인 & end-to-end 하니스 (P3 임계경로 부트스트랩)

> 도구: `tools/bench_solver.py` — 실제 제출 진입점을 40개 인스턴스에 돌려
> `utils.check_feasibility`로 **실제 대회 목적함수**를 측정(패킹 밀도 아님).

## 배경: 팀 상태 점검 결과

리포지토리에 **P1 커밋만** 존재(스케줄러/제출 하니스 부재). 즉 제출 가능한
end-to-end 파이프라인이 없어, 누구의 개선도 점수로 확인 불가 → 임계경로 차단.
P1이 P3 하니스를 부트스트랩해 이를 해소.

## 발견 1: 제공된 baseline_greedy는 제출 불가 수준

`--solver baseline` 측정(전용 CPU, jobs=1):

| 인스턴스 | 시간제한 | 실제 소요 | feasible |
|---|---:|---:|---|
| prob_21 | 15s | **47.6s** | ❌ stage 3 |
| prob_22 | 15s | **21.1s** | ❌ stage 2 |
| prob_23 | 15s | **102.3s** | ❌ stage 2 |
| prob_24 | 15s | **80.8s** | ❌ stage 2 |

❌ **시간제한을 3~7배 초과 + infeasible.** 서버 기준 각 **−1점**(시간초과 AND 비실현).
원인: 보수(repair) 루프의 시간가드가 패스 사이에서만 검사 → 한 패스 내
`check_feasibility`(대형 인스턴스 O(n²) shapely)와 전체 재배치가 예산을 초과.

→ **결론: 패킹 밀도보다 "항상 feasible + 시간 준수"가 압도적 우선.**

## 발견 2: 보장된-feasible·시간안전 솔버 (`solver.py`)

베이당 동시 1블록(시간창 비중첩) → 공간충돌·크레인충돌·순서충돌이 **구조적으로
불가능** → 임의 in-bounds 위치에서 항상 feasible. 병렬머신 리스트 스케줄링(EDD,
가장 빨리 끝나는 베이에 배정)으로 tardiness 완화. O(n·m), 사실상 즉시.

`--solver sequential` 전체 40개 결과:

| 지표 | 값 |
|---|---|
| feasible | **40/40 (100%)** |
| 실행시간 | 인스턴스당 ~0.0s (시간초과 불가능) |
| 총 목적값 | 25,488,584,105 |
| 평균 목적값 | 637,214,603 |
| Σ obj1 (tardiness) | 2,232,300 |
| Σ obj2 (불균형) | 286,049 |
| Σ obj3 (선호) | 275,590 |

✅ **−1 위험 제거 → 양수 점수 floor 확보.** 목적값은 **obj1(tardiness)이 지배적**
(w1≈1.3만~2.9만이라 w1·obj1이 목적값 대부분). 직렬화로 블록이 오래 대기하기 때문.

## 발견 3: 공존 그리디 솔버 (`solver.solve_greedy`) — 4.8배 개선

EDD 순서로 블록을 **가장 빨리 끝나는 베이·시각**에 배치하되, 같은 베이에서
공간적으로 양립 가능하면 **공존**시켜 대기(tardiness)를 줄인다.

**Feasible by construction**: `_feasible_insert`로 검증된 배치만 커밋하므로 보수
루프가 없고 절대 infeasible을 내지 않는다. 이 검사는 check_feasibility의 5단계를
모두 모사한다 — 특히 **같은 시각 진입/퇴출**(stage 5)은 경계를 `<=`로 포함해 양방향
크레인 검사를 하는 보수적·순서독립적 방식으로 처리(strict 부등호만 쓰면 stage 5에서
터지는 버그를 수정). 시간예산(0.9·limit) 초과 시 빈-베이 순차 배치로 안전 폴백 →
시간제한 절대 초과 안 함.

`--solver greedy` 전체 40개 결과 (시간제한 30s):

| 지표 | sequential | **greedy** | 개선 |
|---|---:|---:|---:|
| feasible | 40/40 | **40/40** | 유지 |
| 솔버 시간 | ~0.0s | ~28s (<30s 준수) | — |
| 총 목적값 | 25,488,584,105 | **5,348,307,540** | **4.8× ↓** |
| 평균 목적값 | 637,214,603 | **133,707,688** | 4.8× ↓ |
| Σ obj1 (tardiness) | 2,232,300 | **613,548** | 3.6× ↓ |
| Σ obj2 (불균형) | 286,049 | 175,569 | 1.6× ↓ |
| Σ obj3 (선호) | 275,590 | 138,566 | 2.0× ↓ |

인스턴스별 하이라이트: prob_22 322M→1.8M(180×), prob_21 410M→12.7M(32×).

✅ **현재 제출 후보**: `solver.solve_greedy` (myalgorithm 진입점에서 호출). 100%
feasible, 시간안전, baseline(−1) 대비 확실한 양수 점수.

## 발견 4: 적응형 페이스 deadline — 추가 2배 개선

진단(`stats`): 하드 인스턴스는 **초기 블록이 공존 탐색에 예산을 다 써** 후속 블록이
시간초과로 순차 폴백(prob_38: 250개 중 185개 폴백) → tardiness 폭증. 반면 공존이
끝까지 돌면 탁월(prob_17 300블록 4.3s, obj 0.15M).

해결: 블록당 **누적 페이스 deadline** `t_start + budget·(rank+1)/n`. 빠른 초기 블록이
여유를 은행처럼 적립해 후속 블록이 더 쓰고, 페이스에 뒤처질 때만 공존 탐색을 포기
(공존 불가 블록의 후보 소진을 조기 차단). 예산 내 끝나는 쉬운 인스턴스는 절대
영향 없음(prob_22 obj 불변).

전체 40개 결과 비교 (30s):

| 지표 | greedy(고정) | **greedy(페이스)** | 개선 |
|---|---:|---:|---:|
| feasible | 40/40 | **40/40** | 유지 |
| 총 목적값 | 5,348,307,540 | **2,723,326,004** | **49% ↓ (~2×)** |
| 평균 | 637M→134M | **68,083,150** | 2× |
| Σ obj1 (tardiness) | 613,548 | **301,156** | 51% ↓ |

하드 인스턴스 하이라이트: prob_38 1.66e9→1.06e9, prob_23 90M→46M.

## 발견 5: 시간 견고성 + 병렬 멀티스타트

**시간 견고성** (FAQ: "다양한 시간 제약서 테스트 강력 권장"): 5~60s 전 구간에서
40개 모두 feasible·시간준수. 목적값은 시간에 단조 개선(prob_23 5s 85.5M→60s 22.8M).
→ 서버의 긴 제한(수분~30분)에서 자동으로 더 좋아짐. (`tools/bench_solver.py --timelimit`)

**병렬 멀티스타트** (`solver.solve`, FAQ 권장 4-worker): seed별 EDD 지터로 다양한
스케줄을 4코어 동시 구성 → 목적값 최소 선택. `compute_objective`는 평가서버와 일치
하는 값을 assignments에서 직접(저비용) 계산. feasible-by-construction + 자체 시간제한
+ 단일패스/순차 폴백이라 절대 infeasible·시간초과 없음(격리 샌드박스 검증).

전체 40개 (30s):

| 지표 | greedy(페이스) | **멀티스타트(4코어)** | 개선 |
|---|---:|---:|---:|
| feasible | 40/40 | **40/40** | 유지 |
| 총 목적값 | 2,723,326,004 | **2,346,956,859** | **14% ↓** |
| Σ obj1 | 301,156 | **262,667** | 13% ↓ |

## 발견 6: LNS 탐색 — 큰 인스턴스엔 유효, 4코어 한계로 기본 채택 보류

`solver.solve_lns` 구현: 가장 tardy한 블록 k개 제거 → 재삽입, 개선 시 수락(아니면
O(k) 되돌림). feasible-by-construction + known-good 복원이라 항상 feasible.

측정(30s, construct_frac=0.35):

| 인스턴스 | greedy(full) | LNS | 차이 |
|---|---:|---:|---:|
| prob_39 (250) | 2.52e8 | **1.95e8** | **+22.7%** |
| prob_38 (250) | 1.01e9 | 9.14e8 | +9.2% |
| prob_23 (100) | 2.75e7 | 3.92e7 | **−42.7%** |

- ✅ **큰 인스턴스**: full 구성이 어차피 시간초과 → 빠른 구성+LNS 재배치가 크게 우세.
- ❌ **작은 인스턴스**: full 구성이 이미 강함 → 구성을 줄이면 손해.
- ⚠️ **4코어 한계**: 멀티스타트에 LNS 워커를 섞으면 full-construct **seed 다양성을
  희생**(prob_23이 24M→37M로 퇴보). full-construct 다양성과 LNS를 4코어로 동시
  확보 불가.

→ **결정**: 퇴보를 출시하지 않는다. 기본 entry는 검증된 plain 멀티스타트(2.35e9,
무퇴보) 유지. `solve_lns`는 독립 도구로 보존(`bench_solver.py --solver lns`).
향후 인스턴스 크기 적응형(작으면 full-multistart, 크면 LNS) 또는 코어>4 환경에서
재결합 가치 있음.

## 발견 7: obj3/obj2 폴리시 (선호 베이 재배치, 무퇴보)

**가중 기여도 분석**: 전체로는 w1·obj1이 98.8%지만, **~20개 인스턴스(obj1≈0)는
obj3가 65~100%** 차지. 리더보드는 per-instance라 이들에서 obj3가 직접 점수다.

**원인**: 베이 선택 키가 raw `exit_t` 1순위 → 여러 베이가 납기 전 완료(tardiness 동일
0)면 선호 베이를 버리고 더 이른 종료를 택해 obj3↑.

**실패한 시도**:
- 전역 키를 tardiness-우선으로: obj3 −10% but obj1 +7.5%(베이 늦게 비움→미래
  tardiness↑), 총합 악화.
- 멀티스타트 워커를 키별로 분할(3 exit+1 tard): zero-tardiness 7/11 개선했으나
  exit-seed 다양성 손실로 tardiness 인스턴스 퇴보(집계 +10.7%).

**채택(무퇴보)**: 4 exit 워커(seed 다양성 보존) 유지 + **워커가 일찍 끝나 남는 시간**
(쉬운/zero-tardiness 인스턴스에만 존재; 하드는 예산 소진)에 `_obj3_polish` 실행 —
블록을 더 선호하는 베이로 재배치하되 **정확한 총 목적값이 감소할 때만 수락**
(`_feasible_insert` 검증 + 되돌림). 하드 인스턴스는 폴리시 생략 → 손해 0.

| zero-tardiness 인스턴스 (11개) | exit-only | +폴리시 |
|---|---:|---:|
| 개선 / 퇴보 | — | **10 / 0** |
| 예: prob_2 | 5.13e4 | **4.22e4** (−18%) |
| 예: prob_15 | 1.35e5 | **1.14e5** (−16%) |
| 예: prob_17 | 1.44e5 | **1.23e5** (−14%) |

40/40 feasible·시간준수 유지. (집계 총합은 하드 인스턴스 run간 노이즈에 지배되어
폴리시 효과가 가려지나, 폴리시는 monotone-개선이라 per-instance 무퇴보.)

## 발견 8: 남는 시간 local search (obj1+obj3, 무퇴보)

발견 7의 obj3 폴리시를 일반화(`_local_search`): 기여도(w1·tardiness + w3·선호페널티)
가 큰 블록부터 제거 → **전 베이의 최선 슬롯에 재삽입**, **총 목적값이 엄밀히 감소할
때만 수락**, 개선이 있는 한 라운드 반복. tardiness(obj1)와 선호(obj3)를 동시 공략.
multistart best에서 시작해 개선 move만 받으므로 **per-instance 무퇴보**.

추가로 공존 검사 hoist(발견 직전, `_partition`/`_feasible_pre`)로 entry당 1회만
시간필터링 → 같은 시간에 공존 소폭↑.

전체 40개 (30s):

| 지표 | multistart+hoist | **+local search** | 개선 |
|---|---:|---:|---:|
| feasible | 40/40 | **40/40** | 유지 |
| 총 목적값 | 2.380e9 | **2.193e9** | **−7.8%** |
| Σ obj1 | 267,905 | **244,822** | −9% |

## 발견 9: AABB fast-path (공존 고속화) — 채택

`_feasible_pre`가 검사하는 모든 블록은 시간겹침 집합(coll)의 부분집합임을 증명
(경계 4개 부분집합 ⊆ coll). 따라서 후보 bbox가 coll의 어떤 블록과도 안 겹치면
shapely 없이 즉시 feasible → Block 생성·shapely 생략. 빈 공간 후보가 많아 큰 이득.
40/40 feasible(평가서버 함수 확인), 훈련셋 총 **2.193e9 → 1.953e9 (−11%)**, obj1 −10%.

## 발견 10: 크기 적응형 LNS 디스패치 — 혼재(보류)

AABB 적용 후 재측정: **≥200블록 tardiness-heavy** 인스턴스는 LNS 우세
(prob_31 +21%, prob_33 +10%, prob_37 +12%, prob_38 +8%), **<200**은 multistart 우세
(prob_23 −96%). 그래서 블록수 200 임계로 [full 1 + LNS 3] 디스패치 시도:
- 집계는 −5.4%(1.953e9→1.848e9)였으나, **per-instance 혼재**: prob_9~20처럼
  "블록은 많지만 obj3-지배(tardiness≈0)" 인스턴스 ~14개가 퇴보(LNS가 obj3에 약함).
- **블록 수는 tardiness-지배 여부를 대변하지 못함.** 집계 이득은 소수 대형 tardiness
  인스턴스의 큰 절댓값이 가린 것; 리더보드는 per-instance라 퇴보가 손해.
- → **보류(되돌림).** 올바른 신호는 "대형 AND tardiness-지배"이며, 빠른 obj1 프로브
  (소예산 구성 후 obj1>0 확인)로 구별해야 함. 향후 과제.

현재 제출 권장본: **uniform multistart(4 full) + local search + AABB**(무퇴보).
